1 Overview
The TripXML Web Services are based on OTA specifications 2005A. Amadeus has made a few changes to the original OTA schemas to better fit its business needs. These changes are outlined in this document.
This document provides information on the following web services that are offered by TripXML:
o Travel build: creates a reservation in the provider system
o PNR cancel: cancels a reservation
o PNR read: retrieves a reservation
o Update: add, delete and modify elements in an existing reservation (PNR)
o Queue: provides queue management messages
o Queue read: retrieves a reservation from a specific queue
o Transfer Ownership: changes the ownership of a reservation
o Stored Fare build: create a stored fare record shell in a reservation
The document is based on the OTA messages user guide to help customers familiar with OTA specifications better understand the specific TripXML implementation.
All TripXML web services require the POS group of elements to be included in each request message, as defined below:
<POS>
<Source PseudoCityCode="YOUR PCC">
<RequestorID Type="21" ID="requestor"/>
</Source>
<TPA_Extensions>
<Provider>
<Name>GDS Name</Name>
<System>Target System</System>
<Userid>userid</Userid>
<Password>password</Password>
</Provider>
</TPA_Extensions>
</POS>

The POS data will be provided by the Amadeus support team.

2.2 Definition
The TravelBuild version 3 has the same features as the previous version and a few differences in the response (the request is exactly the same). The response will contain more PNR information especially ticketing data when the PNR was ticketed.

OTA_TravelItineraryRQ definition

OTA did not define any OTA_TravelItineraryRQ message up to now. Because OTA_TravelItineraryRS exists, we have created the equivalent for the request message. Our current version of OTA_TravelItineraryRQ is made of 4 possible XML groups:

o OTA_AirBookRQ
o OTA_VehResRQ
o OTA_HotelResRQ
o TPA_Extensions
The last TPA_Extensions group contains definitions of some new data and some duplicate data already defined in the Air, Car and Hotel book groups. This is to avoid creating, for example, an OTA_AirBookRQ that will contain the names in a Car only PNR.
TPA_Extensions group will typically contain data shared between the three Book XML groups to avoid duplication of redundant data in the request XML message.

2.6 Advance Passenger Information (APIS) Requirement
Advanced Passenger Information (APIS) data for passengers are required by some of the airlines due to government requirements. To support the APIS data to be sent to the requested airlines, IATA approved format are available with TripXML TravelBuild message.

This is supported in TravelBuild message in SpecialServiceRequest element, the format is described below.
Note: TravelerRefNumberRPHList contains the passenger reference at booking creation time. Airline code is sent as YY so the SSR will apply to all airlines in the itinerary.
a) Example with Passport has travel document.

<SpecialServiceRequests>
<SpecialServiceRequest SSRCode="DOCS" TravelerRefNumberRPHList="1">
<Airline Code="YY"/>
<Text>P-GBR-12345678-GBR-30JUN73-M-14JUN12-SMITH-JOHN</Text>
</SpecialServiceRequest>
</SpecialServiceRequests>

P-GBR-12345678-GBR-30JUN73-M-14JUN12-SMITH-JOHN (1)(2)	(3)	(4)	(5)	(6)	(7)	(8)	(9)

1) Travel document type (P = Passport)
2) Travel document issuing country
3) Travel document number
4) Passenger nationality
5) Passenger date of birth
6) Passenger gender
7) Travel document expiry date
8) Surname in travel document
9) Given name in travel document

5 OTA_UpdateRQ/RS

5.1 References
Web service URL: http://amadeusws.tripxml.com/TripXML/wsUpdate.asmx

WSDL: http://amadeusws.tripxml.com/TripXML/wsUpdate?WSDL


5.2 Definition

The Update method allows users to make changes to an existing reservation (PNR) based on the PNR's record locator. The response message returns the full PNR information back.
Note: The response to PNRRead is the same as the response to TravelBuild message.

With the Update method it is possible to:
- insert new elements to the PNR
- delete existing elements from the PNR
- modify existing elements in the PNR
- insert, delete, modify (and any other combination) elements in PNR in one single transaction

When an Update request is received by TripXML, it will:
- retrieve the existing PNR
- apply the insert, delete and modify changes
- end transact the PNR to complete changes
- return the updated PNR back in the response

If any of the Update elements fails (like trying to change an element that does not exist), the whole transaction will fail and no update will take place. An error message specifying the reason for failure will be returned in the response.


5.3 Message structure
Each Position element inside the Update message corresponds to one update action. The Element element contains the update action in the attribute Operation. Operation can be "insert", "delete" or "modify". The Child element contains the element to update. Inside the Element group, the data to be updated is listed.

Delete example: request is to "delete" a Remark in Remarks group:

<Position XPath="OTA_TravelItineraryRS/TravelItinerary/ItineraryInfo/SpecialRequestDetails/Remarks">
<Element Operation="delete" Child="Remarks">
<Remarks>
<Remark>*PR*** AGENCY</Remark>
</Remarks>
</Element>
</Position>

The client application will typically reatrieve the PNR using the PNRRead message and then send the above request to delete the Remark.
The Remarks group as well as the Remark element are identical to the structure they have in the PNRRead response XML.


Insert example: request is to "insert" a Remark in Remarks group:

<Position XPath="OTA_TravelItineraryRQ/OTA_AirBookRQ/TravelerInfo/SpecialReqDetails/Remarks">
<Element Operation="insert" Child="Remarks">
<Remarks>
<Remark>*PR*** TRIPXML AIR BOOKING ENGINE</Remark>
</Remarks>
</Element>
</Position>


The Remarks group as well as the Remark element are identical to the structure they have in the TravelBuild request XML.


Modify example: request is to "modify" a Remark in Remarks group (replace an existing Remark value with a new value)"
<Position XPath="OTA_TravelItineraryRS/TravelItinerary/ItineraryInfo/SpecialRequestDetails/Remarks">
<Element Operation="modify" Child="Remarks"> <!-Remark to modify -->
<Remarks>
<Remark>*PR*** ANOTHER AIR BOOKING ENGINE</Remark>
</Remarks>
</Element>
<Element Operation="modify" Child="Remarks"> <!-Remark to modify with -->
<Remarks>
<Remark>*PR*** TRIPXML AIR BOOKING ENGINE</Remark>
</Remarks>
</Element>
</Position>
The update Remark will show in the PNR response:
<Remarks>
<Remark>*PR*** TRIPXML AIR BOOKING ENGINE</Remark>
</Remarks>

The client application will typically reatrieve the PNR using the PNRRead message and then send the above request to modify the Remark.
The Remarks group as well as the Remark element are identical to the structure they have in the PNRRead response XML.

The first Element group is the one to be modified, the second Element group is the one to be modified with.

6 IssueTicketRQ/RS

6.1 References
Web service URL: http://amadeusws.tripxml.com/TripXML/wsIssueTicket.asmx

WSDL: http://amadeusws.tripxml.com/TripXML/wsIssueTicket?WSDL

6.2 Definition
The Issue Ticket method allows users to request issuing of tickets. With the issue ticket message it is possible to:
o Issue E-ticket alone
o Issue MCO (Miscellaneous Charge Order) alone
o Issue ET and MCO at the same time
o Issue E-ticket and send a notification to the email address stored in the PNR (Amadeus APE element)
o Issue E-ticket and send a notification to the email address stored in the PNR (Amadeus APF element)


6.3 IssueTicket Request message element
Example 1:
Issue e-ticket only (Amadeus TTP/ET entry).

<TT_IssueTicketRQ>
<POS>
<Source PseudoCityCode="JNBZA2222">
<RequestorID Type="21" ID="Amadeus"/>
</Source>
<TPA_Extensions>
<Provider>
<Name>Amadeus</Name>
<System>Training</System>
<Userid>Amadeus</Userid>
<Password>password</Password>
</Provider>
</TPA_Extensions>
</POS>
<UniqueID ID="Z5IG42"/>
<Ticketing TicketType="eTicket"/>
</TT_IssueTicketRQ>


Example 2:
Issue MCO only (Amadeus TTP/TTM entry).
<TT_IssueTicketRQ>
<POS>
<Source PseudoCityCode="JNBZA2222">
<RequestorID Type="21" ID="Amadeus"/>
</Source>

<TPA_Extensions>
<Provider>
<Name>Amadeus</Name>
<System>Training</System>
<Userid>Amadeus</Userid>
<Password>password</Password>
</Provider>
</TPA_Extensions>
</POS>
<UniqueID ID="Z5IG42"/>
<Ticketing IssueMCO="true"/>
</TT_IssueTicketRQ>


Example 3:
Issue e-ticket and MCO (Amadeus TTP/ET/TTM entry).
<TT_IssueTicketRQ>
<POS>
<Source PseudoCityCode="JNBZA2222">
<RequestorID Type="21" ID="Amadeus"/>
</Source>
<TPA_Extensions>
<Provider>
<Name>Amadeus</Name>
<System>Training</System>
<Userid>Amadeus</Userid>
<Password>password</Password>
</Provider>
</TPA_Extensions>
</POS>
<UniqueID ID="Z5IG42"/>
<Ticketing TicketType="eTicket" IssueMCO="true"/>
</TT_IssueTicketRQ>

Example 4:
Issue e-ticket and send an e-ticket receipt to the email address noted in the APE field of the PNR (Amadeus TP/ITR-EMLA/ET entry).
<TT_IssueTicketRQ>

<Source PseudoCityCode="JNBZA2222">

</Ticketing>
</TT_IssueTicketRQ>

The Void Ticket method allows users to void tickets that have been already issued.With the Void ticket message it is possible to void one ticket or multiple tickets in one message.

</Provider>
</TPA_Extensions>
</POS>
<Tickets>
<TicketNumber>001-1234567890</TicketNumber>
</Tickets>
</TT_VoidTicketRQ>

Void multiple tickets.

<TT_VoidTicketRQ Version="1.0">

<Source PseudoCityCode="JNBZA2222">

<TPA_Extensions>
